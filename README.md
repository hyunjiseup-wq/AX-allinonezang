## 0. 시작점 — 올인원장과 원장의 막힌 하루 *(배경)*

**올인원장**([allinonezang.com](https://allinonezang.com))은 동네 소형·중형 학원 원장을 위한 **학원 운영 통합 SaaS**입니다. 수강생 관리, 원비·교재비 자동 청구, 출결, 학부모 알림, 강사 관리를 한 플랫폼에서 처리합니다. 10년 이상 학원을 운영한 현업 원장이 기획에 참여했고, 학원장 커뮤니티 '열피쌤'(회원 3.2만 명)을 기반으로 성장 중이며, 핵심 성장 과제는 **신규 수강생 유치**입니다.

원장은 신기능·이벤트·후기를 네이버 블로그·인스타그램·카카오채널에 홍보하고 싶어 하지만, 홍보에 쓸 시간도 전문 인력도 없습니다. 이 과제는 팀스파르타 AX 해커톤(마케팅/PM 트랙, 3시간, 개인 과제)에서 받은 문제이며, 이 저장소는 그 제출물입니다.

## 1. 진짜 문제는 "복붙"이 아니었다 *(문제)*

원문 문제는 "채널마다 톤·길이가 다른데 원장이 손으로 각각 쓰거나(1건당 1~2시간), 3채널에 똑같이 복붙해 채널 특성이 사라진다"였습니다.

여기서 한 걸음 더 들어가 문제를 **내 언어로 재정의**했습니다.

- 문제는 "글을 대신 써주는 것"이 아니라, **같은 키워드라도 채널별로 다른 심리적 트리거**(블로그=정보 신뢰 / 인스타=감성 공감 / 카카오=즉시 행동)를 **매번 다르게** 담는 것.
- 그리고 그것을 **새 데이터가 들어와도 같은 품질로 재현**할 수 있게 만드는 것.
- 게다가 입력 데이터(25건)에는 **결측·완전중복·표기 흔들림**이 섞여 있어, 콘텐츠를 쓰기 전에 **정제 기준부터 결정**해야 했습니다.

즉 이 과제의 난이도는 "글쓰기"가 아니라 **"판단을 규칙으로 고정해 반복 가능하게 만드는 것"**에 있었습니다.

## 2. 코드가 할 일과 판단이 할 일을 갈랐다 *(해결)*

전체를 하나의 파이프라인으로 묶되, **성격이 다른 두 구간을 명확히 분리**했습니다.

```
새 keywords_input.csv
     │
     ▼
① scripts/clean_keywords.py   ── 완전중복 제거 · tone_hint 정규화 · detail 결측 보완 · type 집계
     │   (결정론적 코드 — 같은 입력엔 항상 같은 출력)
     ▼
② /insight   ── 채널 규칙 + type별 강조 매핑을 케이스마다 적용해 전략 확정
③ /generate  ── 케이스마다 블로그·인스타·카카오 3채널 초안 생성
④ /review    ── 글자 수·톤 차별화 자가 채점, 미달 시 보완
     │   (Claude Code의 "규칙을 따르는 판단" — 문장 품질이 목적)
     ▼
output/blog_draft.md · instagram_draft.md · kakao_draft.md  (24건 × 3채널)
```

- **정제(①)만 순수 코드로 고정**했습니다. 중복 판정·톤 정규화·결측 보완은 "정답이 있는 규칙"이라, 코드로 박아두면 실행할 때마다 동일한 판단이 나옵니다.
- **생성(②~④)은 사람 대신 Claude가 판단하는 구간**으로 남겼습니다. 원장에게 자연스럽게 읽히는 카피는 규칙만으로 품질이 담보되지 않기 때문입니다.
- 그래서 이 파이프라인의 재현성은 **"①의 결정론적 정제 + ②③의 고정된 규칙(스킬 프롬프트)"**에서 나오지, 완성된 문장을 캐싱해서 나오지 않습니다.

과제 제약("별도 API 키 없음 — Claude Code 대화 안에서 직접 생성", "웹 UI 구현 제외") 때문에 별도 생성 앱은 두지 않았고, 자동화는 **스크립트 + 슬래시 커맨드 조합**으로 구현했습니다.

## 3. 가장 오래 붙잡은 두 질문 — "무엇을 쓸까"와 "어떻게 프롬프트를 쓸까" 

이 과제에서 시간을 가장 많이 쓴 곳은 글쓰기 자체가 아니라, 아래 두 질문에 **근거를 붙여 규칙으로 만드는 일**이었습니다.

### 3-1. "무엇을 쓸까" — 채널마다 다르게 쓸 근거를 설계했다

`output/channel-strategy.md`에 채널별 규칙을 **마케팅·원장 관점의 근거와 함께** 명문화했습니다.

| 항목 | 블로그 | 인스타그램 | 카카오채널 |
|---|---|---|---|
| 역할 | SEO 장기 유입·신뢰 | 브랜드 인지·감성 공감 | 즉각 행동 유도·전환 |
| 길이 | 본문 **500자 이상** | 캡션 **150자 이내** | **180자 이내** |
| 해시태그 | 없음 | **5개 고정**(`#올인원장 #학원관리` + 연관 3개) | 없음 |
| 이모지 | 사용 안 함 | 1개 내외 | **1~2개**(3개↑는 스팸) |
| CTA | 부드러운 유도 | "저장해두세요" | 명령형 + **링크 필수** |

- **블로그에 이모지·해시태그를 안 넣는 이유**: 앱을 "검색"해서 들어오는 예비 원장에게는 정보 신뢰도가 우선 — 이모지가 많으면 광고성 글로 읽혀 이탈합니다.
- **인스타 해시태그를 `#올인원장 #학원관리`로 고정하는 이유**: 케이스마다 바뀌는 3개만으로는 검색 유입이 분산됩니다. 고정 2개가 채널 전체를 하나의 해시로 묶어 브랜드 검색 시 누적 노출을 쌓습니다.
- **카카오에 링크를 반드시 넣는 이유**: 업계 자료상 "링크 없이 텍스트만 있으면 전환율이 급락"하기 때문 — 이미 관심 있는 팔로워에게 바로 행동할 통로를 줘야 채널의 존재 이유(전환)가 삽니다.

여기에 **type(신기능/이벤트/후기)별 강조 매핑**과 **`tone_hint` → 채널별 변형표**(예: 전문적인 → 블로그는 수치 중심, 인스타는 "확실히 달라졌어요")까지 붙여, "무엇을, 왜 그렇게" 쓰는지를 근거로 고정했습니다.

### 3-2. "어떻게 프롬프트를 쓸까" — 판단을 스킬 4개에 담아 재현 가능하게 만들었다

매번 즉흥적으로 프롬프트를 치면 결과가 흔들립니다. 그래서 위 규칙을 **Claude Code 스킬(슬래시 커맨드) 4단계에 명문화**해, 새 CSV에도 같은 판단이 재현되게 했습니다.

| 스킬 | 프롬프트에 담은 것 |
|---|---|
| `/analyze` | 정제 스크립트 실행 → 결측·중복·표기 흔들림·type 분포 파악 (전처리 진입점) |
| `/insight` | 채널별 차별화 원칙 + type별 전략 + 예외 처리 방향 확정 |
| `/generate` | 글자 수 규칙·톤 변형 규칙·keyword/brand 반영·예외 처리를 케이스마다 적용해 3채널 생성 |
| `/review` | CLAUDE.md 채점 기준으로 글자 수·톤 차별화 **자가 채점**, 미달 시 보완 |

핵심은 **"정제는 코드(결정론)로, 생성은 규칙을 따르는 판단으로"** 라는 경계를 프롬프트 설계에 그대로 반영한 것입니다. 프롬프트가 규칙집 역할을 하므로, 문장을 캐싱하지 않아도 "새 데이터 → 같은 품질"이 성립합니다.

## 4. 갈림길마다 근거를 남겼다 *(설계 결정)*

주요 판단은 전부 `[02] 올인원장/decisions.md`에 **결정 / 이유 / 버린 대안 / 영향 범위** 형식으로 기록했습니다.

- **완전 중복 행**: 모든 컬럼이 같은 행은 뒤 id를 제거하고 앞 id만 유지 → id 025(=id 001) 제거, **24건**으로 재구성. (같은 걸 두 번 만들면 채널 중복 게시 = 복붙 감점 리스크)
- **tone_hint 정규화**: "긴박" 등 어미 빠진 표기를 표준 톤으로 매핑 → id 017 `긴박 → 긴급한`(다른 긴급 케이스 008·014와 통일).
- **detail 결측**: 빈 셀은 type·keyword 기반 추론 문구로 보완하되 `[추론: …]` 표시를 남겨 원본과 구분 → id 012·020.
- **발행 우선순위 가중치(Challenge)**: 전환 기여도 40% + 반복 빈도 35% + 시즌 적시성 25%. 이 미션이 "신규 수강생 유치"에서 출발했으므로 전환 기여도에 최고 가중치를 부여.
- **제출 직전 QA에서 스스로 잡은 불일치**: 문서화한 규칙과 실제 산출물이 어긋난 곳을 `/review`로 발견해 보완 — 블로그 13건이 (소제목 제외) 500자 미달이었던 것을 본문 보강으로 해결, 인스타 12건에 빠져 있던 `#학원관리`를 채워 넣음. **규칙을 낮추지 않고 산출물을 규칙에 맞췄습니다.**
- **막혔던 지점 & 돌파**: 처음엔 "재현성 = 하나의 앱"이라 여겨 Anthropic API를 직접 호출하는 Streamlit을 만들었지만, 제출 규정("실행 중 외부 AI 호출 앱 금지")과 부딪힌다는 걸 확인. **제출물(스크립트 + 스킬 조합)과 시연 도구(별도 Streamlit)의 역할을 분리**하는 쪽으로 방향을 수정했습니다 — "자동화 = 제출물도 하나의 앱"이라는 전제 자체가 틀렸다는 걸 배운 지점입니다. 시연용 Streamlit은 `demo` 브랜치에서만 배포되며 `.gitignore`로 `main`(제출 대상)에서 제외됩니다.

## 5. "된다"는 말 대신 실측으로 *(결과)*

| 레벨 | 달성 내용 | 만점 |
|---|---|---|
| 🟢 Basic | 24건 × 3채널(총 72개) — 길이 충족 · 톤 차별화 100% · keyword/brand 반영 | 100 |
| 🟡 Standard | 채널 전략 설계(`channel-strategy.md`) + 파이프라인 재현성 실측(`demo/`) | +30 |
| 🔴 Challenge | 키워드 패턴 기반 발행 전략 기획안 2건(P0 95점 / P1 70점) | +30 |

재현성은 말로 주장하지 않고 **눈앞에서 보여줄 수 있는 형태**로 남겼습니다.

- 원본에 없던 keyword·brand 조합의 **더미 CSV 2세트(8건·5건)**를 만들어(매번 다른 완전중복·미등록 톤·결측 패턴) `clean_keywords.py`로 실측 → **처리 결과 100% 일치**.
- 정제 직후 같은 자리에서 끊김 없이 Claude Code 대화로 **신규 7건까지 3채널 생성**(`demo/generated_samples.md`).
- 규칙표에 없던 **미등록 톤 2건**("차분한"→따뜻한 계열, "설레는"→밝은 계열)도 표를 기계적으로 찾지 않고 **문맥에 맞게 판단** — 오탐 없음.

**Before**: 원장이 1건당 채널별로 손수 작성 시 1~2시간(문제 정의서 기준, 실측 아님).
**After**: 24건 × 3채널을 정제 스크립트 + Claude Code 명령 파이프라인으로 **한 세션 내 생성**, 복붙 없이 톤 차별화 100% 준수.

## 6. 숨기지 않은 아쉬움 *(한계)*

- **실제 채널 성과는 측정하지 못했습니다.** Before/After의 시간 수치는 문제 정의서 기준 추정치이지 실측이 아닙니다.
- **채널 API 실연동은 과제 범위 밖**입니다(블로그 자동 발행·카카오 API·인스타 포스팅 제외). 실제 서비스 기능이 되려면 이 연동이 필요합니다.
- **Challenge 기획안을 실제 발행 캘린더(주/월 단위)로 구체화**하지 못했습니다.
- **신기능 8건**(데이터·분석형 / 행정자동화형 / 현장관리형)은 별도 패턴 분석에 포함하지 못했습니다.
- API 자동화 봇은 제출물에는 쓸 수 없어(제약상), 더미 실측 검증과 별개로 **시연 전용 Streamlit**(`demo` 브랜치, 라이브 API 호출)으로 분리해 완성했습니다.

## 7. 새 CSV로 똑같이 돌려보기 *(재현 방법)*

```bash
# 0) 정제 — 결정론적 스크립트 (같은 CSV엔 항상 같은 결과)
python "[02] 올인원장/scripts/clean_keywords.py" "[02] 올인원장/data/keywords_input.csv"
#   → data/keywords_input.cleaned.csv + data/cleaning_report.md 생성

# 1) 이어서 Claude Code 대화에서 순서대로 실행
/analyze    # 정제 결과·데이터 현황 파악
/insight    # 채널·type별 전략 확정
/generate   # 24건 × 3채널 초안 생성 → output/
/review     # 글자 수·톤 차별화 자가 채점, 미달 시 보완
```

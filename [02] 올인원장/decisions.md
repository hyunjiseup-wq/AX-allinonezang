- 대안으로 고려했던 것: 두 id 모두에 대해 콘텐츠를 생성하되 동일하게 생성 — 실익이 없어 기각

## [Standard] tone_hint 표기 정규화 규칙
- 결정 내용: "긴박" 등 어미가 빠지거나 표기가 흔들린 값을 사전 정의된 표준 톤(전문적인/따뜻한/긴급한/자랑스러운/밝은/친근한)으로 매핑. 사전에 없는 값은 어미를 뗀 어간으로 재시도 후 그래도 못 찾으면 원본 유지
- 선택한 이유: 같은 의미의 톤이 표기만 달라 다른 톤으로 분류되면 채널 전략(강조 문구·CTA)이 일관되지 않게 적용될 위험이 있음
- 대안으로 고려했던 것: 결측·흔들림 값을 참가자가 매번 수동으로 확인 — 새 CSV가 들어올 때마다 반복 작업이라 스크립트로 고정
- 영향 범위: id 017 "긴박" → "긴급한" (다른 긴급 케이스 008·014와 표기 통일)

## [Standard] detail 결측 보완 규칙

## [/review 자가 점검] 제출 직전 QA — 블로그 글자 수·인스타 해시태그 불일치 발견 및 보완

## [시연] 제출물과 시연 도구의 역할 분리
- 결정 내용: Anthropic API를 실시간 호출해 3채널 초안을 생성하는 Streamlit 앱을 만들되, 제출 대상인
  `main` 브랜치에는 포함하지 않고 별도 `demo` 브랜치에서만 관리·배포한다. `.gitignore`의
  `시연용-스트림릿/` 규칙으로 `main`에서는 자동 제외되고, `demo` 브랜치에서는 그 규칙을 명시적으로
  덮어써(`git add -f`) 포함시킨다.
- 선택한 이유: CLAUDE.md 제약("실행 중 외부 AI 호출 앱 금지", "대시보드·웹 UI 구현 제외")은 제출물에
  적용되는 규정이지 시연 방식 전체를 막는 게 아니라고 판단. 실제로 해커톤 현장 매니저에게 시연 목적의
  실시간 API 호출은 확인받음. "자동화 = 제출물도 하나의 앱"이라는 전제를 버리고, 제출용(스크립트+스킬)과
  시연용(Streamlit) 두 트랙으로 분리.
- 대안으로 고려했던 것: 시연용 앱을 아예 포기하고 Claude Code 대화창 전환만으로 시연 — 청중 입장에서
  화면 전환이 부자연스러워 기각.
- 영향 범위: `demo` 브랜치 전체, README "막혔던 지점 & 돌파"·"아쉬움" 항목, 발표자료 11장(회복 01).

## [배포] Streamlit Cloud 대괄호 경로 버그 발견 및 우회
- 결정 내용: 시연용 앱의 원래 경로 `[02] 올인원장/시연용-스트림릿/app.py`로 Streamlit Cloud에 배포 시
  "Error installing requirements" 발생. 로그 확인 결과 `requirements.txt` 경로에서 `[02] ` 접두사가
  통째로 사라진 채 탐색되고 있었음(`[02]`를 글롭 브래킷 패턴으로 오인). `demo` 브랜치에서만 앱을
  저장소 최상위(`시연용-스트림릿/`)로 옮기고, `app.py` 내부에서는 `Path`로 `[02] 올인원장`을 명시적으로
  가리키도록 수정해 우회.
- 선택한 이유: 대괄호 있는 폴더명(`[02] 올인원장`)은 과제 제출 규정상 유지해야 하는 이름이라 바꿀 수
  없음. Python 코드 내부의 `Path` 조합은 대괄호를 문자 그대로 다루므로 문제없고, Streamlit Cloud의
  자체 경로 탐색 로직만 문제였기 때문에 배포 경로만 옮기는 쪽이 가장 안전.
- 대안으로 고려했던 것: `[02] 올인원장` 폴더 자체의 이름을 바꾸는 방법 — 제출 구조와 다른 참가자 관례를
  깨서 기각.
- 영향 범위: `demo` 브랜치의 `시연용-스트림릿/app.py` 내부 `PROJECT_DIR` 경로, Streamlit Cloud
  "Main file path" 설정값.
